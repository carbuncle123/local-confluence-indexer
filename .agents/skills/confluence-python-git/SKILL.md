# Confluence Python Git

この skill は、このリポジトリで日常的に実装作業を進めるときに使う。

## 最初に見るもの

- `AGENTS.md` を読む。
- `confluence_local_sync_index_spec.md` を仕様の正本として扱う。
- 挙動を変える前に、記憶で進めず現在の作業に対応する仕様書の節を開いて確認する。

## 基本ワークフロー

1. `git status` と現在のファイル構成を確認する。
2. 実装に入る前に、その作業専用のブランチを毎回新しく切る。
3. 最小で意味のある実装単位を決める。
4. その単位に必要なファイルだけを編集する。
5. 触ったコードに対して絞った検証を実行する。
6. コミットを整理し、PR を作る前提で変更内容を要約する。

## Python の方針

- Python 3.11 以上を前提にする。
- `tools/` 配下では、小さく役割のはっきりしたモジュールを保つ。
- 状態を持つ設計が明確に有利でない限り、深いクラス階層より明示的な補助関数を優先する。
- 公開関数や主要データ構造では、読みやすさが上がる範囲で型ヒントを付ける。
- ファイル操作には `pathlib.Path` を優先する。
- retry、timeout、logging はスクリプトごとに重複させず、なるべく共通化する。
- 秘密情報や生の Authorization header を出力しない。

## このプロジェクト固有の注意

- ローカル同期コンテンツやインデックスは生成物であり、コミットしない。
- sync 状態は成功時のみ進める。
- 差分同期は保守的に扱い、取りこぼしリスクより安全な再取得オーバーラップを優先する。
- 検索ツールはリポジトリルートから簡単に実行できる状態を保つ。

## Git の方針

- 実装ごとに毎回新しいブランチを作る。
- 差分は焦点を絞り、コミットは小さく保つ。
- 作業完了時は毎回 PR を作る前提で、説明しやすい単位に変更をまとめる。
- 無関係なユーザー変更は巻き戻さない。
- 生成されるローカルパスを増やしたら、すぐに `.gitignore` を更新する。
- 省略した検証があれば、次に何を確認すべきか分かるように明記する。
